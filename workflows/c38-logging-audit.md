# C38 Logging Audit

Re-audits the codebase to discover all file types generated by the C38 controller, vehicle, and phone,
then reconciles findings against the existing Notion database at:
https://app.notion.com/p/skydio/C38-Logging-Audit-372dc5ce82938078966bfc76e320c5f1

## When to run
- After adding a new file kind to `FileKindEnum`
- After adding a new upload provider (e.g. new `ControllerXxxProvider`)
- After changing `allowed_filekind_prefixes` in appcore params
- After changing `ucon_default_kind_configs` in FlightDeck params
- After modifying the vehicle AFU, C38 SOC AFU, or Kotlin upload paths
- Periodic sanity check (~quarterly)

## Step 1: Discover vehicle-side files exposed via FlightDeck UCON

Search these files to find all file kinds the vehicle exposes through `/api/ucon/available`:

1. **Default always-uploaded kinds**: Read `vehicle/infrastructure/flight_deck/params.py` and find `ucon_default_kind_configs`. These are always uploaded regardless of cloud config.
2. **Per-flight file gatherers**: Read `vehicle/infrastructure/flight_deck/utils/ucon_utils.py` and find `FIND_FLIGHT_FILE_BY_KIND`. Each entry maps a `FileKindEnum` to a function that finds files for a flight.
3. **Non-flight file gatherers**: In the same file, find `FIND_ALL_FILES_OF_KIND`. Each entry maps a `FileKindEnum` to a function that finds global (non-flight) files.
4. **Available files assembly**: Read `vehicle/infrastructure/flight_deck/utils/ucon_api_impl.py`, function `handle_available_files_request()`. This combines on-demand, per-flight, and non-flight files into the response.
5. **Upload config cloud overrides**: Read `tools/cloud_api/utils/ucon_kind_config.py` to see which kinds are added by cloud config groups (telemetry, automatic_file_upload, etc.).

## Step 2: Discover which uploaders consume FlightDeck files

There are 3 uploaders that poll `/api/ucon/available`. Check each for filtering:

1. **Vehicle AFU** (Path A): `vehicle/automatic_file_upload/executables/vehicle_automatic_file_upload_front.py` â€” uploads everything FlightDeck returns (no kind filtering). Gated by `vehicle_upload_enabled` in cloud config.
2. **Appcore ucon** (Path B): `mobile/shared/appcore/params.py` â€” find `allowed_filekind_prefixes` list. Only kinds matching these prefixes are uploaded. Also check `hidden_filekinds` for UI visibility.
3. **Kotlin mobile upload** (Path C): `mobile/android/android-shared-skydio/features/logs/` â€” user-initiated only. Check `LogUploadRepository`, `CloudFileUploader`, `UconFileUploader`, and `LogDownloadRepository` for what files are downloaded from the vehicle and uploaded.

## Step 3: Discover C38 controller-specific upload paths

1. **C38 SOC AFU** (Path D): Read `util/ucon/platforms/controller/executables/automatic_file_upload_front.cc`. Find which providers are instantiated (e.g. `ControllerAnalyticsProvider`, `ControllerProcessLogsProvider`).
2. **Controller analytics providers**: Read `util/ucon/platforms/controller/analytics_provider.cc` (SOC) and check for radio analytics (NFS path).
3. **Controller process logs provider**: Read `util/ucon/platforms/controller/process_logs_provider.cc` for on-demand export behavior.

## Step 4: Discover live telemetry / Datadog paths

1. **SkyCat**: Read `accessories/controller_analytics/skycat_front.cc` and `skycat_front.h` for event forwarding to Datadog and analytics.
2. **DropBoxMonitor**: Read `mobile/android/skycat/src/main/java/com/skydio/skycat/DropBoxMonitor.kt` for Android system event capture.
3. **MetricsCollector**: Read `mobile/android/skycat/src/main/java/com/skydio/skycat/MetricsCollector.kt` for device metrics polling.
4. **Escape hatch**: Read `util/ucon/platforms/controller/ESCAPE_HATCH.md` for AFU critical event forwarding.

## Step 5: Identify gaps

Check for data sources that exist on the device but have NO upload path:

- `/data/tombstones/` (SOC native crash dumps)
- `/data/anr/` (SOC ANR traces)
- `/data/system/dropbox/` (SOC DropBox raw files)
- `/data/misc/logd/` (SOC persisted logcat ring buffer)
- `/sys/fs/pstore/` on SOC (kernel panic logs â€” compare with Radio's `S01logrotate`)
- Radio live `dmesg` (kernel ring buffer)
- Radio raw process logs at `/mnt/logs/process_logs/` (no auto-upload, only manual)
- Phone-side appcore logs (REPORT_STATUS output)
- `/data/vendor/skycat/spillover/` (transient, recovered on restart only)
- SELinux audit logs (`avc: denied` in logcat/dmesg)

## Step 6: Update the Notion database

The database is: `C38 File Upload Registry`
- Database ID: `03ca1ac033354a5f938d6a010d67ede6`
- Data source ID: `fc23f005-faf4-490f-856d-fddc4cc6ecc2`

Schema (all entries must have these properties):
- **File Type** (title): Human-readable name
- **Kind** (text): FileKindEnum string, or "N/A" for non-file-based entries
- **Origin** (select): Vehicle | C38 SOC | C38 Radio | Phone
- **Category** (select): Vehicle FlightDeck | Controller | Phone-local | Live Telemetry | Gap
- **Upload Status** (select): Automatic | Cloud-Config Gated | User-Initiated Only | Partial Coverage | Not Uploaded
- **Upload Paths** (multi_select): A: Vehicle AFU | B: Appcore ucon | C: Kotlin manual | D: C38 SOC AFU | E: Datadog
- **Trigger** (select): Always-on default | Cloud-config driven | User-initiated | On-demand | Automatic (live) | N/A
- **Source Details** (text): How the file is gathered (function name, path, mechanism)
- **Notes** (text): Risk notes, partial coverage explanation
- **Ref #** (number): Sequential reference number

### Upload Status rules:
- **Automatic**: At least one always-on automatic upload path (default kind config or always-running provider)
- **Cloud-Config Gated**: Automatic but requires org opt-in via cloud config
- **User-Initiated Only**: Only uploaded when a user explicitly triggers it
- **Partial Coverage**: Some indirect coverage exists (e.g. included in a tarball, or summary captured) but raw files not uploaded
- **Not Uploaded**: No upload path exists

### How to reconcile:
1. Fetch the existing database entries using `notion-query-data-sources`
2. For each discovered file type, check if a matching row exists (match on Kind + Origin)
3. If it exists, verify all properties are still accurate â€” update if changed
4. If it's new, create a new page with the next Ref #
5. If an existing entry no longer exists in code, add a note but do NOT delete it (it may be relevant for historical context)
6. After reconciliation, verify the Coverage Overview chart still renders correctly
